## Goal Objective

按照 TDD 和测试金字塔原则，实现 RepoAssure v0.3 的分发与修复闭环就绪能力。

目标是在不破坏 v0.2 accepted 验收证据、CLI/MCP 兼容入口、本地优先隐私边界和现有 artifact contract 的前提下，让 RepoAssure 可以通过本地 CLI、MCP 和 GitHub Action wrapper 形成可重复的 hardening -> repair handoff -> validation-only -> patch plan 闭环。

## 前置条件

启动本 goal 前，必须先完成 `monorepo readiness audit`：

- 确认当前 repo 是可运行的分阶段 monorepo，而不是成熟完成态 monorepo。
- 明确哪些 monorepo 缺口属于 v0.3 P0：GitHub Action wrapper、examples、repo hygiene / public-release readiness checks、repair loop contract。
- 明确哪些 monorepo 缺口不阻塞 v0.3：`packages/core` 抽取、app shell 深迁移、benchmark package ownership、`apps/web-dashboard`。
- 保持 `packages/core` 仅在降低 v0.3 分发或 repair loop 风险时才迁移；否则不把结构迁移扩张为 v0.3 scope。

## Authorization

用户授权 Codex 全自动执行本 goal，包括：

- 修改 TypeScript/JavaScript 源码、测试、GitHub Action wrapper、文档和验收材料。
- 新增或调整 CLI/MCP/GitHub Action 分发示例。
- 强化 repair task package、repair handoff、repair execution report 和 patch plan 的 agent-consumption contract。
- 更新 goal audit、README、user guide、testing strategy、architecture docs、release checklist、dev log 和 blockers。
- 按 TDD 运行 unit、integration、E2E、acceptance、goal audit、typecheck、lint、build 和必要的 repo hygiene。
- 在需要本地端口监听、Playwright Chromium 或 GitHub API 权限时按环境规则请求提权。

仍必须遵守：

- 不默认修改目标 repo 业务源码。
- 不自动创建 Git commit、branch、issue、PR、advisory 或 release。
- 不上传目标 repo 源码、日志、截图、trace、env value、token、cookie 或私有 artifact。
- GitHub Action wrapper 必须调用本地 CLI，不依赖 hosted RepoAssure 服务。
- Hosted dashboard、Team Cloud、远程 artifact storage 和 auto-fix mode 必须先经过单独 ADR。
- 遇到无法解决的问题，必须记录到 `docs/logs/blockers.md`。

## TDD Plan

1. Red：为 GitHub Action wrapper 添加结构测试和 fixture invocation 期望。
2. Green：实现最小 Action wrapper，复用本地 CLI，不上传私有内容。
3. Red：为 repair task package / handoff / execution report / patch plan 增加 agent-consumption contract 测试。
4. Green：补齐稳定字段、验证命令、证据引用和 prompt 边界。
5. Red：为 public release readiness 添加可执行 checklist 或 repo hygiene 扩展测试。
6. Green：实现 license/dependency/secret/artifact release readiness 检查的最小可运行版本。
7. Refactor：在测试保护下整理 `packages/core` 相关边界，只在降低分发或 repair loop 风险时迁移代码。
8. Final：刷新 README、architecture、acceptance、testing、strategy、goal audit 和 dev log，并运行完整质量门禁。

## Success Criteria

| 类别 | 通过标准 |
| --- | --- |
| 分发 | GitHub Action wrapper 可在 fixture 或 CI 中调用 RepoAssure CLI |
| 本地优先 | Action 和文档明确不上传目标 repo 内容，不依赖 hosted service |
| AI IDE 消费 | repair task package、handoff、execution report、patch plan 对 agent 有稳定入口和字段 |
| 修复闭环 | 至少一个 fixture 或真实 repo 可完成 handoff -> validation-only -> patch-plan |
| 安全 | 新增输出继续脱敏敏感值，repo hygiene 防止 generated/private artifacts 入库 |
| 文档 | v0.3 spec、README、architecture、testing、acceptance、release checklist 与实现一致 |
| 门禁 | repo hygiene、unit、typecheck、lint、build、goal audit 通过；需要本地监听/浏览器的完整门禁按 blocker 规则提权或记录 |

- Agent contract：`repair-handoff-package.json`、`repair-execution-report.json` 和 `patch-plan.json` 生成 `agentContract`，包含 schema、read order、next commands、result semantics 和 no-auto-edit boundaries。

## Non-goals

- 不实现默认 auto-fix。
- 不创建 PR。
- 不发布 npm package。
- 不添加仓库级 `LICENSE`，除非 public release checklist 完成并另行授权。
- 不实现 hosted dashboard、Team Cloud、SSO/RBAC 或 enterprise policy center。
