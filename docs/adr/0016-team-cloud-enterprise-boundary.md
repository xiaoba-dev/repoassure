# ADR-0016: Team Cloud and Enterprise Commercial Edition Boundary

- Status: Accepted
- Date: 2026-06-25
- Deciders: hardening-mcp maintainers

## Context

ADR-0009 establishes an open-core commercialization model: the local CLI, MCP integration, artifact contracts, acceptance modes, repair plans, and local-first run bundles belong in the open core, while hosted dashboards, collaboration, enterprise integrations, and governance can become commercial product surfaces.

The public release readiness work now makes this boundary more urgent. RepoAssure can publish an adoption-friendly local core, but future Team Cloud and Enterprise surfaces need a sharper decision before implementation. Without that decision, hosted work could accidentally weaken the local-first promise, fork artifact schemas, or imply that users must upload target repo source to receive value.

This ADR records the Team Cloud and Enterprise commercial edition boundary before building cloud runtime code.

## Decision

Define Team Cloud and Enterprise as commercial packaging layers over the open artifact contract, not as replacements for local-first execution.

Team Cloud may store, index, compare, visualize, comment on, and route RepoAssure artifacts generated by the open core. Enterprise may add private deployment, SSO/RBAC, policy management, audit retention, approvals, compliance reports, and integrations with code hosts, ticketing systems, SIEM/GRC tools, and enterprise AI platforms.

The commercial editions must follow these boundaries:

- Keep CLI, MCP, GitHub Action, acceptance modes, repair artifacts, and artifact schemas in the open core unless a later ADR changes the open-core boundary.
- Treat hosted dashboard, team collaboration, enterprise integrations, and advanced governance as commercial packaging layers over those artifacts.
- Do not implement hosted paid features in this increment; this increment defines the roadmap, architecture boundary, and acceptance criteria only.
- No target repo source upload by default. Commercial products may ingest generated RepoAssure artifacts, explicit user-provided metadata, and opt-in sanitized attachments, but must not silently collect repository source, logs, screenshots, traces, env values, or private artifacts.
- Commercial control-plane work must not fork the artifact contract. If Team Cloud needs new fields, they should first be added to the open artifact schema or recorded as commercial metadata outside the open artifact payload.
- Enterprise/on-prem packaging must be deployable without requiring public internet access to customer code repositories.

The first planning artifacts for this decision are:

- `docs/product/specs/team-cloud-enterprise-spec-v0.1.md`
- `docs/architecture/specs/team-cloud-enterprise-architecture-v0.1.md`

## Consequences

### Positive

- Preserves RepoAssure's local-first adoption path while creating a clear paid product roadmap.
- Gives future cloud and enterprise work a concrete privacy and artifact-contract boundary.
- Helps buyers understand that commercial value comes from history, collaboration, governance, and integrations rather than withholding local execution.
- Reduces the risk that a hosted dashboard becomes a cloud-only hardening workflow.

### Negative

- The commercial edition cannot rely on source-code lock-in; it must win through workflow value, governance, retention, and trust.
- Some advanced analytics may require careful schema evolution before they can be shipped commercially.
- Team Cloud and Enterprise implementation work remains blocked until the spec and architecture boundary are accepted and separately prioritized.

### Follow-up

- Use `docs/product/specs/team-cloud-enterprise-spec-v0.1.md` as the commercial roadmap source for hosted dashboard, team collaboration, enterprise integrations, and advanced governance.
- Use `docs/architecture/specs/team-cloud-enterprise-architecture-v0.1.md` as the implementation boundary before adding hosted services, cloud storage, organization accounts, or enterprise deployment packages.
- Add executable checks for any future cloud metadata or artifact ingestion code before implementation.
- Revisit ADR-0009 and this ADR if pricing, deployment, or customer requirements pressure the project to move open artifact contracts into the commercial layer.
