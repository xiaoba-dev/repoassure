# hardening-mcp 用户访谈脚本

最后更新：2026年6月18日
状态：草案
关联规格：

## TL;DR

本访谈脚本用于验证：`hardening-mcp` MVP 是否能为使用 AI IDE 构建 Web 应用的独立开发者和 AI 交付工作室，解决真实、高频且有商业价值的问题。

访谈时不要一开始就推销产品。应先从用户近期真实行为、失败经历、当前替代方案以及为更好工作流付费的意愿开始。

## 研究目标

验证或证伪以下用户与商业假设：

| ID | 假设 | 需要了解什么 |
| --- | --- | --- |
| U1 | 目标用户会发布或交付 AI 生成的 Web 应用 | 他们使用 AI coding tools 构建和发布应用的频率 |
| U2 | Vibe-coded 应用带来痛苦的上线就绪不确定性 | 什么会坏、坏得多频繁、代价有多高 |
| U3 | 现有替代方案缓慢、不可靠或经常被跳过 | 他们今天在上线或交付前如何检查 |
| U4 | 本地优先的 AI IDE 工具适合自然工作流 | MCP/CLI 嵌入编码环境是否顺手 |
| U5 | 报告、测试和补丁 diff 是有价值的输出 | 哪类输出最能激发使用意愿 |
| U6 | 存在早期付费意愿 | 用户是否愿意付费、试用或提供真实 repo |

## 目标受访者

优先完成 10 次访谈，覆盖两类人群：

| 细分人群 | 目标数量 | 画像 |
| --- | ---: | --- |
| 独立开发者 | 5 | 最近使用 Cursor、Codex、Claude Code、Windsurf、Lovable、Bolt、v0 或类似工具构建过 Web 应用 |
| AI 交付工作室 | 5 | 向客户交付 AI 生成应用的自由职业者、代理机构或小团队 |

可选的后续人群：

| 细分人群 | 为什么后续再做 |
| --- | --- |
| 接管 AI 原型的初创团队工程师 | 对 V1 有价值，但 MVP 首先面向 maker 和工作室 |

## 筛选题

排期前使用以下问题筛选。

### 必须满足的条件

如果受访者至少对以下 3 项回答“是”，即可继续：

- 你是否在过去 30 天内使用 AI coding tool 构建过 Web 应用？
- 你是否尝试发布、演示、销售或交付该应用？
- 你是否使用过 Next.js、React、Vite、Node.js、Supabase、Firebase 或类似技术栈？
- 你在上线前是否遇到过 bug、流程损坏、缺少测试、部署问题或不确定性？
- 你是否愿意展示一个真实 repo、匿名化 repo 或最近的失败案例？

### 排除条件

如出现以下情况，应跳过或降低优先级：

- 他们只用 AI 写小代码片段，而不是构建应用。
- 他们近期没有构建过 Web 应用。
- 他们无法描述近期的具体失败案例。
- 他们当前只做企业 Java/.NET/移动端。
- 他们想要完整 QA 外包服务，而不是开发者工具。

## 访谈形式

| 项目 | 建议 |
| --- | --- |
| 时长 | 30 分钟 |
| 形式 | 优先视频会议 |
| 录制 | 先征得同意 |
| Demo | 在行为问题之后再演示 |
| 激励 | 可选礼品卡或免费 early access |
| 素材请求 | 访谈后请求 repo、失败截图、测试报告或上线检查清单 |

## 开场话术

开场时使用：

```text
感谢你抽时间交流。我正在探索一个面向使用 AI coding tools 构建应用的开发者工具。今天不是销售访谈，我想了解 AI 生成应用接近上线或客户交付时，真实发生了什么。

我会问一些关于近期真实项目、哪里出过问题、你如何检查质量，以及你会为什么付费的问题。没有标准答案。具体案例比泛泛观点更有价值。
```

## 访谈流程

| 环节 | 时间 | 目标 |
| --- | ---: | --- |
| 背景 | 5 分钟 | 了解受访者和近期 AI coding 工作流 |
| 近期项目 | 7 分钟 | 锚定一个具体应用或交付案例 |
| 失败与替代方案 | 8 分钟 | 理解痛点、代价和当前替代方案 |
| 概念反馈 | 6 分钟 | 测试 MVP 定位和输出价值 |
| 定价与下一步 | 4 分钟 | 测试付费意愿和试点意愿 |

## 第 1 部分：背景

提问：

1. 你通常构建哪类应用？
2. 过去 30 天内你使用过哪些 AI coding tools？
3. 你最近的应用中，大概有多少是由 AI 生成或修改的？
4. 你通常是为自己、用户、客户还是内部团队构建应用？
5. 你最常使用的技术栈是什么？

记录：

| 字段 | 备注 |
| --- | --- |
| 细分人群 | 独立开发者或 AI 交付工作室 |
| AI tools | Cursor、Codex、Claude Code 等 |
| 技术栈 | Next.js、React、Vite、Node 等 |
| 构建频率 | 每月构建的应用数量 |
| 上线/交付频率 | 每月上线或客户交付次数 |

## 第 2 部分：近期项目

锚定一个具体项目。

提问：

1. 回想最近一个你主要或部分用 AI 构建的 Web 应用。它是什么？
2. 在你认为它“准备好了”之前，它必须完成什么？
3. 风险最高的用户路径是什么？
4. 你如何检查它是否正常工作？
5. 你有测试吗？如果有，是什么类型？如果没有，为什么没有？
6. 上线前检查花了多久？

有效追问：

- 能不能带我走一遍你的上线或交付检查清单？
- 你手动点击检查了哪些路径？
- 哪些地方你信任 AI 处理？
- 哪些地方你不信任？

## 第 3 部分：失败与替代方案

要求对方给出具体失败案例。

提问：

1. 上线前什么地方坏了，或差点坏了？
2. 上线、演示或客户交付后，什么地方坏了？
3. 你是如何发现这个问题的？
4. 复现和修复花了多久？
5. AI tool 是否在某个时刻让问题变得更糟？
6. 之后你是否添加了回归测试？
7. 今天你如何避免类似问题再次发生？

重点追问以下类别：

| 类别 | 示例追问 |
| --- | --- |
| 启动 | 应用是否在另一台机器或部署环境中无法运行？ |
| 路由 | 页面是否白屏或崩溃？ |
| 表单 | 登录、checkout 或创建流程是否失败？ |
| API | API routes 是否意外返回 4xx/5xx？ |
| Env vars | 缺失 key 或本地配置是否阻塞过项目？ |
| AI 回归 | Agent 是否修好了一个问题，却破坏了另一个功能？ |
| 测试 | 测试是否缺失、不稳定或被忽略？ |

## 第 4 部分：概念反馈

只在行为问题之后，简要介绍 MVP。

概念话术：

```text
MVP 设想是一个面向 AI IDE 用户的本地优先 MCP Server 和 CLI。你在自己的 Web App repo 里运行它。它会分析技术栈、启动应用、使用 Playwright 模拟用户路径、捕获 console/network/runtime 故障、生成 Playwright 测试，并写出一份硬化报告，包含复现步骤、证据、就绪度评分和补丁 diff。

它从 Next.js、React、Vite 和 Node 应用开始。它不会上传你的代码，也不会试图成为完整的 SaaS QA 平台。
```

提问：

1. 这个工具会放在你当前工作流的哪个位置？
2. 哪类输出最有价值：报告、测试、补丁 diff、就绪度评分，还是截图/trace？
3. 哪类输出你会忽略？
4. 作为本地 MCP Server、CLI、GitHub App 或 SaaS，你最信任哪种形态？
5. 什么会让你运行一次后停止使用？
6. 什么会让你在每个项目上都运行它？

## 第 5 部分：定价与承诺

直接但中立地提问。

提问：

1. 如果这个工具能帮你避免一次上线失败或客户尴尬，这件事值多少钱？
2. 你是否愿意把它作为个人工具付费？如果愿意，什么月费区间合理？
3. 如果你是工作室，你更愿意按 repo、按 seat，还是按客户交付项目付费？
4. 你是否愿意在未来两周内用一个真实 repo 试用？
5. 你是否愿意介绍另一个有同样问题的人给我们？

建议测试的价格锚点：

| 细分人群 | 价格锚点 |
| --- | --- |
| 独立开发者 | $15-$30/月 |
| Power user | $49-$99/月 |
| AI 交付工作室 | $99-$299/月或按项目计费 |
| 后续团队版 | $20-$50/seat/月 |

不要过度依赖精确价格。相比随口说“我会付费”，愿意提供真实 repo 是更强信号。

## 强信号标准

每次访谈按每个维度 0 到 2 分评分。

| 维度 | 0 | 1 | 2 |
| --- | --- | --- | --- |
| 频率 | 很少构建 AI 应用 | 偶尔构建 | 每周或重复构建 AI 应用 |
| 痛点 | 轻微不便 | 明显延误或焦虑 | 损失时间、金钱、客户信任或上线信心 |
| 当前替代方案 | 不需要替代方案 | 手动点击或临时检查 | 明确但痛苦的 QA/checklist/test 流程 |
| MVP 匹配度 | 工作流匹配不清楚 | 如果容易会尝试 | 现在就想放进 AI IDE 或 CLI |
| 输出价值 | 泛泛报告就够 | 测试或报告有用 | 测试、证据和补丁 diff 是必需品 |
| 承诺 | 礼貌性兴趣 | 想看 demo | 提供 repo、介绍、试点或付款 |

最高分：12。

解读：

| 分数 | 含义 | 动作 |
| --- | --- | --- |
| 0-4 | 信号弱 | 不作为核心 persona 证据 |
| 5-8 | 信号混合 | 提取异议并细化范围 |
| 9-12 | 信号强 | 可作为试点用户或设计伙伴候选 |

## 模式复盘

10 次访谈后，总结：

| 问题 | 决策输入 |
| --- | --- |
| 哪个细分人群痛点更强？ | 为 PRD v0.2 确定优先 persona |
| 哪类故障最常见？ | 决定优先探索/检查模块 |
| 哪类输出最重要？ | 决定优先报告/测试/补丁 diff |
| 哪种入口最自然？ | MCP vs CLI vs GitHub App |
| 什么阻碍信任？ | 证据、隐私、准确性、不稳定测试 |
| 谁愿意提供真实 repo？ | 招募 benchmark/试点用户 |

## 通过标准

如果满足以下条件，用户发现轨道通过：

- 完成至少 10 次访谈。
- 至少 5 位受访者评分达到 9 分或以上。
- 至少 3 位受访者提供真实或匿名化 repo 进行测试。
- 至少 2 位受访者表现出明确付费或试点意愿。
- 可以识别出最强细分人群。
- 前 3 类故障模式足够清晰，可指导 PRD v0.2。

## 失败或转向标准

如果出现以下情况，应重新考虑 MVP：

- 大多数用户认为该问题低频。
- 用户偏好手动 QA 或现有工具，且没有紧迫感。
- 用户不希望在工作流中使用本地 MCP/CLI。
- 相比上线就绪测试，用户更看重广泛架构清理。
- 用户不愿提供真实 repo 或失败案例。
- 付费意愿模糊，没有人承诺试点。

## 推荐招募渠道

优先寻找已经处在 AI coding 工作流中的用户：

- Cursor、Claude Code、Codex、Windsurf、Lovable、Bolt 和 v0 社区
- Indie hacker 群体
- AI builder Discords
- 在 X/Twitter 发布帖子，征集近期 AI 生成应用失败案例
- 自由职业者和代理机构社区
- 现有人脉中正在交付 AI-built MVP 的开发者
